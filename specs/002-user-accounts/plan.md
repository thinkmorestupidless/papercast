# Implementation Plan: User Accounts with Auth0 Authentication

**Branch**: `002-user-accounts` | **Date**: 2026-03-18 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/002-user-accounts/spec.md`

## Summary

Add Auth0 JWT authentication to all HTTP endpoints using the Akka SDK's built-in `@JWT` annotation, and introduce a `UserAccountEntity` (Key Value Entity) to store per-user saved queries. Users are identified by the `sub` claim from the validated JWT. The existing `ResearchPodcastEndpoint` is updated to require authentication. A new `UserAccountEndpoint` provides CRUD for saved queries and a "run from saved query" action.

## Technical Context

**Language/Version**: Java 21
**Primary Dependencies**: Akka SDK 3.5.16 (no new external dependencies required)
**Storage**: Akka Key Value Entity (in-memory / Akka persistence — same as existing entities)
**Testing**: JUnit 5, Akka TestKit, `KeyValueEntityTestKit` for unit tests, `TestKitSupport` for integration tests
**Target Platform**: JVM / Akka platform (local dev + cloud deploy)
**Project Type**: Akka web service (extending existing service)
**Performance Goals**: Auth overhead < 50ms per request (local JWT validation, no Auth0 network call per request)
**Constraints**: JWT validation done locally via Akka's key configuration; no Auth0 SDK dependency
**Scale/Scope**: Per-user saved query lists; no pagination required at this scale

## Component Design

### UserAccount (domain record)

```java
// src/main/java/com/example/domain/UserAccount.java
public record UserAccount(String userId, List<SavedQuery> savedQueries) {
    public record SavedQuery(String id, String queryText, Instant savedAt) {}

    public static UserAccount create(String userId) { ... }
    public UserAccount withQuerySaved(String queryText) { ... }   // deduplicates by queryText
    public UserAccount withQueryDeleted(String queryId) { ... }   // throws if not found
    public boolean hasQuery(String queryText) { ... }
    public Optional<SavedQuery> findQuery(String queryId) { ... }
}
```

### UserAccountEntity (Key Value Entity)

```java
// src/main/java/com/example/application/UserAccountEntity.java
@Component(id = "user-account")
public class UserAccountEntity extends KeyValueEntity<UserAccount> {

    public record SaveQuery(String queryText) {}
    public record DeleteQuery(String queryId) {}
    public record SavedQueryResponse(String id, String queryText, String savedAt) {}
    public record SavedQueriesResponse(List<SavedQueryResponse> queries) {}

    public Effect<SavedQueryResponse> saveQuery(SaveQuery command)
    public Effect<Done> deleteQuery(DeleteQuery command)
    public Effect<SavedQueriesResponse> getQueries()
}
```

**Entity key**: Auth0 `sub` claim (passed from endpoint as path/context).

### UserAccountEndpoint (HTTP Endpoint)

```java
// src/main/java/com/example/api/UserAccountEndpoint.java
@HttpEndpoint("/account")
@Acl(allow = @Acl.Matcher(principal = Acl.Principal.INTERNET))
@JWT(validate = JWT.JwtMethodMode.BEARER_TOKEN)
public class UserAccountEndpoint extends AbstractHttpEndpoint {

    @Post("/queries")   saveQuery(SaveQueryRequest)         → 200 SavedQueryResponse
    @Get("/queries")    getQueries()                        → 200 SavedQueriesResponse
    @Delete("/queries/{queryId}")  deleteQuery(String queryId) → 200 | 404
    @Post("/queries/{queryId}/run") runQuery(String queryId) → 201 CreatePodcastResponse | 404
}
```

User identity extracted in every handler: `String userId = requestContext().getJwtClaims().subject().get()`

## Test Strategy

### Unit Tests — `UserAccountEntityTest.java`

Uses `KeyValueEntityTestKit<UserAccount, UserAccountEntity>`:
- Save query → state contains query
- Save duplicate query → state unchanged (idempotent)
- Save blank query → error returned
- Delete query → query removed from state
- Delete non-existent query → error returned
- Get queries on new entity → empty list

### Integration Tests — `UserAccountEndpointIntegrationTest.java`

Uses `httpClient` + `bearerTokenWith(Map.of("sub", userId))`:
- No token → 401 on all endpoints
- Valid token → save, list, delete, run flow
- Cross-user isolation: user A cannot see user B's queries (404 on delete/run)
- Run saved query → 201 with workflowId
- Run non-existent query → 404

## Configuration

No changes to `application.conf` required for local development. The Akka dev key handles JWT validation locally.

For production deployment, Auth0 RS256 key must be registered via CLI (see `quickstart.md`).

## Deployment Notes

After deployment, register the Auth0 public key:
```bash
akka secrets create asymmetric auth0-public-key --public-key ./auth0.pem
akka services jwts add research-podcast-creator \
  --key-id auth0 --algorithm RS256 \
  --issuer https://YOUR_DOMAIN.auth0.com/ \
  --secret auth0-public-key
```
